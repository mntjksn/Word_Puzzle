# ㅎㅏㄴㄱㅡㄹ (Word Puzzle)
Unity 기반 한글 단어 자모 추리 퍼즐 게임입니다.
한글 음절을 초성·중성·종성 단위로 분해해 "숫자야구(스트라이크/볼/아웃)" 방식으로 정답을 맞히는 게임으로, 싱글 플레이·일일 도전·Photon 기반 실시간 1:1 멀티플레이 세 가지 모드를 제공합니다.
Firebase로 유저 인증과 클라우드 데이터 동기화를 처리했습니다.

## 프로젝트 개요
- 개발 환경 : Unity (Mobile)
- 사용 언어 : C#
- 개발 기간 : 2026.06 ~ 2026.07
- 개발 인원 : 1인 (개인 프로젝트)
- 플랫폼 : Android (모바일, 세로 화면 전용)
- 기술 스택 : Photon PUN2 · Firebase · TextMeshPro · JSON

## 주요 기능
- 싱글 플레이 — 글자 수 선택 후 자모 단위 숫자야구 판정, 포인트로 힌트 구매, 중도 이탈 시 진행 상태 자동 저장 후 이어하기
- 일일 도전 — 날짜를 시드로 모든 유저에게 동일한 단어 제공, 1일 1회 도전 제한, 스트릭 기록
- 멀티플레이 — Photon 기반 1:1 실시간 대전, 턴제 제출 · 60초 턴 타이머, 턴 수에 따른 자동 힌트 공개, 상대에게만 결과를 숨기는 히든 아이템
- 자모 분해 판정 엔진 — 한글 음절을 초성/중성/종성으로 분해하고 쌍자음·겹받침·복합모음까지 개별 자모로 나눠 Strike/Ball/Out 판정
- 클라우드 저장 시스템 — PlayerPrefs 로컬 저장을 기본으로, Firebase 익명 인증 기반 기기 간 데이터 동기화 지원
- 인게임 튜토리얼 — 판정 규칙, 자모 치환 규칙, 모드별 진행 방식을 안내하는 "게임 방법" 팝업

## 기술적 고민
### 1. 한글 자모 분해 판정 로직 설계
- 한글 음절을 유니코드 연산으로 초성/중성/종성 인덱스로 분해하고, 쌍자음·겹받침·복합모음은 규칙표 기반으로 개별 자모까지 추가 분해해 판정 정확도를 높였습니다.
- 판정에 쓰이는 토큰과 화면에 보여지는 토큰을 분리해서, 화면에는 회전 치환 규칙(ㅗ·ㅜ·ㅓ·ㅏ → ㅏ 계열)을 적용한 자모를 보여주면서도 실제 정답 판정은 원본 자모 기준으로 정확히 이뤄지도록 구조를 나눴습니다.
- 이 로직을 싱글/일일/멀티 세 모드가 그대로 공유하는 순수 로직 계층(Core)으로 분리해 모드별 중복 구현을 없앴습니다.

### 2. Photon 기반 턴제 멀티플레이 동기화
- Room Custom Properties(wordId, turnIndex, turnStartTime)를 기준으로 양쪽 클라이언트의 턴 상태를 맞추고, RaiseEvent로 제출 · 턴 종료 · 게임 종료 이벤트를 전달했습니다.
- 단어 설정 이벤트와 턴 설정 이벤트가 거의 동시에 도착할 때, 단어 DB 로드가 늦으면 턴 처리가 통째로 누락되는 문제가 있어 두 이벤트를 독립적으로 처리하도록 구조를 재정리했습니다.
- 타이머 만료 직후 제출 이벤트가 도착하면 짧은 순간 상태 플래그가 어긋나 제출이 무시되는 race condition을 발견해, 판정 조건을 이벤트 데이터 자체(턴 번호 · 액터 번호) 기준으로 바꿔 해결했습니다.
- 히든 아이템(상대에게만 결과 숨김) 기능을 위해 같은 제출 데이터를 정상 버전과 히든 버전으로 나눠 대상 플레이어별로 다른 RaiseEvent를 전송하도록 구현했습니다.

### 3. 저장 데이터 호환성과 UI 레이아웃 안정성
- JsonUtility + PlayerPrefs 기반 저장 구조에서 필드 삭제·이름 변경 시 기존 유저 데이터가 깨질 수 있다는 점을 고려해, 필드는 추가만 하고 기존 필드는 유지하는 방향으로 저장 스키마를 관리했습니다.
- Unity UI Layout Group(Vertical/Horizontal Layout Group)의 `childControlHeight` 설정에 따라 자식 오브젝트의 실제 크기가 프리팹에 남은 값으로 잘못 계산되는 문제를 발견해, 레이아웃 그룹이 자식 크기를 직접 제어하도록 설정을 통일해 해결했습니다.

## 회고
### 주요 어려움
  #### 1. 멀티플레이 이벤트 처리 순서 문제
  - Room Property 갱신과 제출 이벤트가 예상한 순서대로 도착하지 않는 상황에서 힌트 표시나 제출 판정이 꼬이는 문제가 반복됐고, 이벤트 도착 순서에 의존하지 않고 각 이벤트가 가진 데이터(턴 번호, 액터 번호)로 유효성을 직접 검증하는 방식으로 해결했습니다.
  #### 2. 자모 판정 로직의 예외 케이스 처리
  - 쌍자음·겹받침·복합모음처럼 한 글자가 여러 자모로 쪼개지는 케이스를 판정 로직에 빠짐없이 반영하는 것이 까다로웠고, 규칙표 기반으로 분해 로직을 분리하면서 정리했습니다.
  #### 3. UI 레이아웃 컴포넌트 조합 문제
  - Layout Group 여러 개를 중첩해서 쓸 때 특정 설정 조합에서 텍스트가 화면에 아예 렌더링되지 않는 문제를 겪었고, 각 레이아웃 컴포넌트가 실제로 자식의 크기를 제어하는지 하나씩 확인하며 원인을 좁혀 해결했습니다.
### 배운 점
- 여러 모드(싱글/일일/멀티)가 같은 핵심 로직을 공유할 때는 그 로직을 순수 함수 계층으로 완전히 분리해두는 것이 유지보수에 훨씬 유리하다는 것을 배웠습니다.
- 실시간 멀티플레이에서는 "이벤트가 순서대로 도착한다"는 가정을 하지 않고, 각 이벤트가 스스로 자신의 유효성을 검증할 수 있도록 설계해야 한다는 것을 경험했습니다.
- 저장 데이터 구조를 처음부터 확장 가능하게 설계해두면 이후 기능이 추가되어도 기존 유저 데이터를 안전하게 유지할 수 있다는 것을 체감했습니다.
